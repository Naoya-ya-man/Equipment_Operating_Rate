# 要件定義書 - 製造業における設備稼働率監視ボード

## 1. インテント分析サマリー

- **ユーザーリクエスト**: `要件定義.md` をベースに、設備稼働率監視ボードの開発をAI-DLCで開始する
- **リクエストタイプ**: 新規プロジェクト（New Project）
- **スコープ見積もり**: マルチコンポーネント（Multiple Components）— CSV疑似データ生成、SSMS(SQL Server)へのMERGE取り込み、稼働率算出、Streamlit WEB表示、定期実行自動化の5つの要素で構成
- **複雑度見積もり**: 中〜やや複雑（Moderate）— 各コンポーネント自体は仕様が明確だが、コンポーネント間の連携（ファイル移動、DB整合性、定期実行）に注意が必要

## 2. アーキテクチャ上の決定事項（質問への回答に基づく）

| 項目 | 決定内容 |
|---|---|
| 定期実行方式 | GitHub Actions **Self-hosted Runner** をローカルPCに構築し、GitHub Actionsのワークフローから12時間毎に実行する（ローカルSQL Server Expressへ到達可能） |
| WEB画面公開方式 | **第一段階はローカルPCでの `streamlit run` 稼働を優先**。将来的にStreamlit Community Cloud等への公開URL発行を検討する（クラウド公開時のDB到達性は別途対応が必要な既知の課題として記録） |
| DB認証方式 | **Windows統合認証（Trusted Connection）**。ユーザー名・パスワードは不要なため、認証情報を秘匿する必要はない |
| Python環境 | **venv + pip + requirements.txt** |
| GitHubリポジトリ | **既存のリポジトリが存在**（詳細は後日共有される想定。現時点ではローカルディレクトリで開発を進める） |

## 3. 機能要件（Functional Requirements）

### FR-1: CSV疑似データ生成（データ転送前）
- `データ転送前/` フォルダに `A1_ProcessingMachine_Utilization_Rate.csv` を作成する
- カラムはSSMSテーブル定義の順序に合わせる: `Product_Number, Machine_Name, Machine_Number, Processing_Start_Time, Processing_Completion_Time, Sum_DateTime, Pass_judgment`
- **生産フロー（重要）**: 1つの製品はA→B→C→D→Eの順に5台の加工機タイプを通過し、Eの加工が完了した時点で完成品となる。各工程の完了ごとに1行のレコードを生成する（同一品番ベースに対し、A〜E分で合計5行のレコードが作られる）
  - 稼働率の算出（FR-3）は号機ごとの実加工時間ベースであり、同一品番が複数工程を経由すること自体は稼働率計算に影響しない
- **品番(Product_Number)**: 7桁。先頭6桁は製品を一意に識別するベース番号、最後の1桁はその行を処理した加工機のアルファベット（例: ベース番号が同じ製品でも、Aで処理された行は末尾`A`、Bで処理された行は末尾`B`となる）
- **加工機名称(Machine_Name)**: A/B/C/D/Eのいずれか（そのレコードの工程を表す）
- **加工号機(Machine_Number)**: 各加工機タイプにつき1〜5号機からランダム選択
- **加工時間の基準（標準加工時間 ±10分程度のランダム変動）**:
  - 加工機A: 約20分
  - 加工機B: 約30分
  - 加工機C: 約40分
  - 加工機D: 約25分
  - 加工機E: 約15分
  - ごく低確率で標準時間の2〜3倍のデータ（設備停止相当）を許容する
- **開始/終了時刻・工程間の整合性**:
  - 各工程の開始/終了時刻を生成し、その差分秒数を `Sum_DateTime` (INT) に格納する
  - 同一製品において、後工程（例:B）の開始時刻は前工程（A）の終了時刻以降になるよう時系列の整合性を保つ（A完了 → B開始 → B完了 → C開始 → … → E完了で完成品）
- **不良判定(Pass_judgment)**: 低確率で "NG"、基本は "OK"
- **稼働日カレンダー考慮**:
  - 日曜日: 全休 → CSV生成をスキップ
  - 月曜日: 0:00〜9:00は未稼働として除外
  - 休憩時間帯は加工時間に含めない: 10:00-10:20 / 12:15-13:15 / 15:00-15:15（日勤）、18:00-18:45 / 21:00-21:15（遅番）、2:00-2:45 / 5:00-5:15（夜勤）
- **1日の理論生産数（完成品ベース）に基づくランダム投入**: 固定値（1日80台）は使用せず、稼働可能時間から時間的に矛盾しない生産数を都度計算する
  1. 加工機タイプ(A〜E)ごとに「1号機あたりの理論処理可能数 = その日の稼働可能秒数 ÷ 平均加工時間」を算出し、号機数（1〜5）を掛けて加工機タイプごとの1日の理論上限を求める
  2. 最も加工時間が長い工程（ボトルネック）の理論上限を、その日全体の完成品数の理論上限として採用する。基準値（A=20分,B=30分,C=40分,D=25分,E=15分、5号機ずつ）で試算すると、ボトルネックは加工機Cとなり、概算値は以下の通り:
     - **通常稼働日**（休憩時間215分を差し引いた稼働可能時間 約73,500秒/日）: C = 73,500秒 ÷ 2,400秒 ≒ 30台/号機 × 5号機 ≒ **1日あたり約150台が理論上限**
     - **月曜日**（0:00〜9:00の非稼働＋日中休憩を差し引いた稼働可能時間 約44,700秒/日）: C = 44,700秒 ÷ 2,400秒 ≒ 18台/号機 × 5号機 ≒ **約90台が理論上限**
     - **日曜日**: 全休のため0台（CSV生成スキップ）
     - これらはあくまで「矛盾しない上限」の目安であり、実行スクリプトは対象日の実際の稼働可能秒数（曜日・月曜補正・休憩時間を反映）から都度この計算を行う
  3. 実際に生成する完成品数は、上記の理論上限を超えない範囲で、低確率の設備停止相当データ（2〜3倍の加工時間）による実効稼働率の低下も加味し、理論上限よりやや少なめの範囲でランダムに決定する
  4. 各加工機タイプ内の1〜5号機はまんべんなく（偏りなく）稼働するようランダム化し、特定号機に割り当てが偏らないようにする

### FR-2: CSV → SSMS 取り込み（MERGE処理）
- CSVの各カラムをSSMSのデータ型に合わせて型変換する
- `[dbo].[A1_ProcessingMachine_Utilization_Rate]` テーブルへ **MERGE文** で取り込む
- MERGE条件: 同一の `Product_Number` + `Machine_Number` の組み合わせが既に存在する場合はINSERTしない（重複防止）
- 参照負荷軽減のため、MERGE条件に使うカラム（`Product_Number`, `Machine_Number` 等）にインデックスを設定する
- 取り込み完了後、処理済みCSVを `データ転送前/` → `データ転送後/` へ移動する
- 取り込み中にエラーが発生した場合、該当CSVを `エラー項目/` へ移動する

### FR-3: 稼働率算出
SSMSのデータを参照し、以下を算出する:
1. **日別稼働率**: 号機ごとに `その日の実加工時間(秒) ÷ 1日の計画時間(86400秒) × 100`
2. **週間稼働率**: 号機ごとに `週間(月曜〜日曜)の実加工時間合計 ÷ 週間計画時間合計 × 100`
3. **週間想定稼働率**: 号機ごとに `週間加工枚数 × 標準加工時間 ÷ 週間計画時間合計 × 100`（標準加工時間はFR-1の基準値を使用）
4. **月別稼働率**: 日別/週別集計を月単位に積み上げて算出
- 計画時間の補正: 三交代制のため基本は86400秒/日。日曜日は全休（計画時間0）、月曜日は9:00開始のため差し引いた時間を計画時間とする
- **週の起算日（UOW-3 Functional Designで確認・確定）**: 「週」は月曜日始まり〜日曜日終わりのISO週とする

### FR-4: WEB画面表示（Streamlit）
- BIツールの「モーションボード」を意識したダッシュボード構成
- 画面内に以下3つの表・グラフを表示:
  1. 日別稼働率（折れ線グラフ、号機ごとに色分け）
  2. 週別稼働率と週間想定稼働率（2系統を1つの表/グラフに表示。想定稼働率は破線で表示）
  3. 月別稼働率（表）
- 号機（5台）ごとにクリックで表示/非表示を切り替えられるボタンUIを設ける
- 折れ線グラフにカーソルを合わせると、その時点の稼働率の値がツールチップで表示される

### FR-5: 定期実行の自動化
- GitHub Actions（Self-hosted Runner経由）で12時間毎に以下の順で自動実行する:
  1. CSV生成 → SSMS取り込みのPythonスクリプト（FR-1, FR-2）
  2. 稼働率算出・WEB表示更新のPythonスクリプト（FR-3, FR-4に必要なデータ更新）
- ワークフロー定義（`.yml`）はリポジトリの `.github/workflows/` に配置する

## 4. 非機能要件（Non-Functional Requirements）

- **NFR-1（パフォーマンス）**: MERGE処理・参照クエリの負荷軽減のため、SSMS側に適切なインデックス（`Product_Number`, `Machine_Number` 等の複合インデックス）を設計する
- **NFR-2（セキュリティ）**: DB接続はWindows統合認証を使用するため、シークレット管理は不要（ただし将来SQL認証に変更する場合は `.env` / GitHub Secretsで管理する方針とする）
- **NFR-3（可用性/自動化）**: Self-hosted Runnerがローカル常時稼働している必要がある（PCの起動状態に依存）
- **NFR-4（保守性）**: CSV生成・DB取り込み・稼働率算出・WEB表示を独立したPythonスクリプト/モジュールとして分離し、それぞれ単体でも実行・テスト可能にする
- **NFR-5（データ整合性）**: 処理済み/エラー時のファイル移動（データ転送後・エラー項目）により、取り込み状況をファイルシステム上で追跡可能にする

## 5. 前提・仮定（低影響のため質問せず仮定として進める事項）

- タイムゾーンはすべて日本時間（JST）とする
- 月別稼働率の算出に必要な過去データは、SSMSに蓄積される全期間を対象とし、WEB画面側で表示期間の絞り込みは行わない（将来的な拡張候補とする）
- 完成品の生産数は固定値（1日80台）を廃止し、FR-1の理論生産数計算（ボトルネック工程基準、約150台/通常日・約90台/月曜）に基づいて日々算出する
- ~~`Pass judgment` カラム名の半角スペースはPython/SQL側では `Pass_judgment` として扱う（要件定義書のカラム名表記ゆれに対する解釈）~~ → **【訂正】実SSMS環境での検証の結果、この仮定は誤りと判明**。実際のテーブルの列名は要件定義書の原文どおり `[Pass judgment]`（半角スペースあり）だった。SQL文中の列参照はこれに合わせて修正済み（Python側の内部属性名 `pass_judgment` はそのまま維持）
- Streamlit Community Cloud等への公開時のDB到達性問題は、現段階ではスコープ外（既知の将来課題としてOPERATIONSフェーズ以降で扱う）
